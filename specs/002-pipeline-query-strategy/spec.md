# Feature Specification: 流水线查询策略优化

**Feature Branch**: `002-pipeline-query-strategy`

**Created**: 2026-06-08

**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 分模块文档生成（Priority: P1）

流水线运营人员希望每类技术文档（架构、数据模型、业务流程等）只基于与其直接相关的源码查询结果生成，而非将所有查询数据混合后统一生成，从而提升每类文档的准确性和聚焦度。

**Why this priority**: 这是本次优化的核心价值——输入更聚焦，生成质量更高，也更容易定位哪类文档出了问题。

**Independent Test**: 对同一代码库分别运行旧方案和新方案，验证新方案中每类文档仅引用其对应领域的符号，不出现跨领域符号污染。

**Acceptance Scenarios**:

1. **Given** 含有服务层、数据层、接口层的代码库，**When** 执行流水线，**Then** `business-flows.md` 只描述业务方法流程，不含数据实体字段定义；`data-model.md` 只描述实体结构，不含接口路由信息。
2. **Given** 某领域（如接口层）在源码中不存在，**When** 执行流水线，**Then** 对应文档（`api-docs.md`）标注"无可用源码证据"并跳过，质量报告记录该领域缺失，不计入完整性分母。
3. **Given** 流水线完成分模块生成，**When** 质量门禁检查，**Then** `doc_completeness_score` 按各模块是否有实质内容计算，不含标注为"无源码证据"的领域。

---

### User Story 2 — 查询范围自适应（Priority: P1）

流水线运营人员希望采集阶段的查询范围能根据项目实际规模自动调整，避免小型项目浪费查询配额、大型项目因单次上限导致信息截断。

**Why this priority**: 当前硬编码的查询文件数上限对不同规模项目均不最优，是信息丢失的根本原因。

**Independent Test**: 对小型（<50 文件）和大型（>200 文件）代码库各运行一次，验证两种规模下文档完整性得分均 ≥ 0.8。

**Acceptance Scenarios**:

1. **Given** 用户未指定查询配额参数，**When** 流水线开始，**Then** 先查询项目总文件数，根据规模自动决定每次查询的批次大小。
2. **Given** 某领域文件数超过单批上限，**When** 采集该领域，**Then** 自动循环查询并合并结果，直至全部文件覆盖或达到最大批次数（默认 5 批）。
3. **Given** 用户通过参数手动指定批次大小，**When** 流水线执行，**Then** 使用手动值，日志中注明"使用手动配置"。

---

### User Story 3 — 查询覆盖率可见（Priority: P2）

流水线运营人员希望在质量报告中看到每个领域的查询覆盖情况，以便判断文档的可信度。

**Why this priority**: 不透明的查询范围导致用户无法判断文档是否完整，覆盖率是信任度的基础。

**Independent Test**: 运行完成后，`quality.json` 中包含每个领域的 `queried_files` 和 `total_files` 字段。

**Acceptance Scenarios**:

1. **Given** 流水线完成采集，**When** 写入质量报告，**Then** `quality.json` 中每类文档对应一条覆盖记录，含已查询文件数和估算总文件数。
2. **Given** 某领域覆盖率低于 80%，**When** 汇总报告展示，**Then** 该领域标注覆盖不足警告，提示用户该文档可信度受限。

---

### Edge Cases

- 某领域完全不存在（如纯前端项目无后端服务层）：跳过该领域文档生成，质量报告标注"无源码证据"，不计入完整性分母。
- 循环查询时某批次返回空结果：终止该领域循环，记录实际已覆盖文件数，不无限重试。
- 用户指定批次大小超过项目总文件数：按实际文件数执行单次查询，不报错。
- 某模块文档生成失败：记录失败原因到质量报告，继续生成其余模块，不中断整体流水线。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 流水线必须将每个领域的查询与对应文档生成绑定为局部闭环：该领域查询完成后立即生成对应文档，不跨领域混合输入。
- **FR-002**: 流水线必须在采集开始前查询项目总文件数，并根据结果自动决定每个领域的查询批次大小。
- **FR-003**: 当某领域文件数超过单批上限时，流水线必须自动循环查询并合并结果，直至全部文件覆盖或达到最大批次数上限（默认 5 批）。
- **FR-004**: 流水线必须支持通过运行参数手动指定查询批次大小，手动值优先于自动计算值。
- **FR-005**: 流水线必须在质量报告中为每类文档记录该领域的已查询文件数和估算总文件数。
- **FR-006**: 当某领域覆盖率低于 80% 时，流水线必须在汇总报告中对该领域标注覆盖不足警告。
- **FR-007**: 当某领域源码不存在时，流水线必须跳过该领域文档生成，在质量报告中标注"无源码证据"，并将该领域从完整性分母中排除。
- **FR-008**: 文档完整性评分必须基于"有实质内容的已生成文档数 / 应生成文档数"计算，应生成文档数不含标注为"无源码证据"的领域。

### Key Entities

- **领域查询结果**：单次或多次循环查询后合并的源码片段集合，与某一文档类型（架构/服务/数据/接口/状态/异常）一一对应。
- **批次配置**：控制每次查询返回文件数的参数，包含自动计算值和用户手动覆盖值。
- **覆盖记录**：质量报告中每个领域的 `queried_files` / `total_files` 条目，反映本次运行的查询完整性。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对同一代码库，新方案各类文档中出现跨领域无关内容的比例相比旧方案下降 ≥ 50%（人工抽查 10 条文档段落）。
- **SC-002**: 对总文件数超过 100 的代码库，流水线文档完整性得分相比旧方案提升 ≥ 0.1。
- **SC-003**: 运行完成后，`quality.json` 中每类已生成文档均有对应覆盖率字段，缺失率为 0。
- **SC-004**: 对小于 50 个文件的项目，端到端运行时间不因分模块和循环查询增加超过 20%。

## Assumptions

- 代码库已完成索引，流水线无需处理索引建立。
- 各领域的文件数估算通过文件列表过滤获得，允许 ±20% 误差。
- 最大循环批次数默认为 5，超限后记录警告并使用已获取的部分结果。
- 分模块生成不改变现有质量门禁逻辑（完整性阈值仍为 0.8）。
- 本次优化范围限于流水线的采集和文档生成步骤，不涉及 SRS 转译、UML 生成和 refine loop。
