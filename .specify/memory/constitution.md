# 逆向工程 Skill 套件章程

## 核心原则

### 一、四阶段流水线完整性

本套件中的每个 Skill 必须归属四个流水线阶段之一，且不得跨阶段边界：

- **阶段 0 — 文档生成**：通过 spec kit 对目标系统生成完整技术文档，确保文档质量达标后方可进入下一阶段（参见原则六）。
- **阶段 1 — 采集**：通过 codegraph/SQLite 工具从异构源码中提取结构。
- **阶段 2 — 转译**：将技术事实转换为技术无关的业务语义（SRS 层）。
- **阶段 3 — 生成**：从 SRS 中间文档生成目标平台代码。

每个 Skill 必须在其 frontmatter 中声明所属阶段。SRS 文档是阶段间唯一允许的交接产物；任何阶段不得从下游阶段消费原始源码。

### 二、技术无关 SRS 层

SRS 中间文档（srs.md）不得包含：编程语言名、框架名、注解/装饰器名、类名/方法名或数据库技术术语。

技术事实必须按以下标准映射进行转译：

| 代码事实 | 业务语义输出 |
|---------|------------|
| 参数有必填校验 | 该字段为必填项 |
| 有权限校验注解/中间件 | 需要"[业务权限名]"权限 |
| 有事务注解/装饰器 | 操作具有原子性保证（NFR-可靠性）|
| 有缓存注解/配置 | 查询结果有 N 分钟有效缓存（NFR-性能）|
| 有软删除字段 | 删除操作为逻辑删除，数据可恢复 |
| 有乐观锁/版本字段 | 并发修改时系统自动检测冲突 |
| 有状态枚举 | 触发业务状态图生成 |

违反本原则时，转译阶段 Skill 必须停止执行并报告违规术语。

### 三、分层输出结构

Skill 必须以严格分离的两层结构输出结果：

- **第一层（技术无关）**：`srs.md` — FR 编号体系（FR-001+）、NFR 五类表格、
  业务视角 UML（使用业务状态名的状态图、业务类图）。
- **第二层（技术参考）**：`hla.md` + `uml/technical/` — 技术调用链、
  组件依赖关系、通信协议。

第一层产物必须自成体系，下游代码生成无需访问第二层。

### 四、UML 图覆盖要求

Skill 必须按指定优先级生成所有图表：

| 优先级 | 图类型 | Mermaid 语法 | 输出路径 |
|--------|--------|-------------|---------|
| 必须 | 流程图 | `flowchart TD` | `flowcharts/` |
| 必须 | 序列图 | `sequenceDiagram` | `flowcharts/` |
| 必须 | ER 图 | `erDiagram` | `database.md` |
| P1 | 业务状态图 | `stateDiagram-v2` | `uml/business/` |
| P1 | 业务类图 | `classDiagram` | `uml/business/` |
| P1 | 技术组件图 | `graph LR` | `uml/technical/` |

Skill 不得将必须生成的图表标记为可选或延期，除非有文档化的说明理由。

### 五、Skill 自主进化

每个 Skill 必须通过定义好的反馈循环支持自主改进：

- 每个 Skill 必须暴露 `self-eval` 入口，根据 SRS 质量评估标准（完整性、技术无关性、UML 覆盖率）对上次运行结果打分。
- 分数低于阈值时，必须在输出最终结果前触发 `refine`（精化）流程。
- Skill 必须将精化决策（修改内容及原因）记录至 `.codebook/skill-evolution.log`。
- 当 Skill 检测到系统性缺陷时，可向自身 prompt 模板提出修订建议，将提案写入 `.codebook/proposals/<skill-name>-YYYY-MM-DD.md` 供人工审查。
- 任何 Skill 不得在未经人工审批提案的情况下修改其他 Skill 的源文件。

### 六、文档先行原则

在对任意代码库执行逆向工程之前，必须先用 spec kit 生成该系统的完整技术文档：

- **自描述标准**：技术文档必须达到"无需查阅源码即可理解系统行为"的程度。
- **可还原性**：熟练工程师仅凭 `文档 + codegraph MCP` 应能完整还原代码实现，不得存在文档中未记载的关键逻辑。
- **文档范围**：系统架构、数据模型、业务流程、接口契约、状态机、错误处理策略、依赖约束均需入档。
- **流水线调整**：标准逆向工程流程由"采集 → 转译"扩展为"文档生成 → 采集 → 转译"，文档阶段不得绕过。
- **质量门槛**：文档质量未达标的系统不得进入采集阶段；质量评估标准参见 `quality.json` 中的文档完整性指标。

### 七、Codegraph 优先提取

阶段 1 的 Skill 必须以 codegraph MCP 工具作为主要提取机制：

- 使用 `codegraph_files` / `codegraph_search` / `codegraph_callers` / `codegraph_node` 提取结构。
- LLM 语义推断必须基于 `source_text` 字段操作，而非直接读取原始文件，以确保分块和缓存行为的一致性。
- 向量库（Qdrant）集成为可选项，仅限 Phase E（SRS 后期增强）；不得阻塞或门控更早的流水线阶段。
- codegraph 项目地址：https://github.com/colbymchenry/codegraph

## Skill 流水线标准

本套件中的所有 Skill 必须：

- 在 YAML frontmatter 中声明 `stage`、`inputs` 和 `outputs`。
- 在执行前验证所需的上游产物是否存在。
- 以原子方式写入输出（临时文件 → 重命名），避免写入不完整导致的数据损坏。
- 在每个主要输出旁附带可机器读取的质量报告（`quality.json`）。
- 在没有完整流水线的情况下也能独立运行和独立测试。

## 治理

本章程优先于本仓库中的所有其他规范。修订需满足以下要求：

1. 在 `.codebook/proposals/` 中提交书面提案，说明变更内容及理由。
2. 经过审查并获得明确批准后，方可修改章程文件。
3. 按语义化版本规则（如下定义的 MAJOR/MINOR/PATCH）进行版本升级。
4. 在同一次提交中同步更新所有依赖模板。

**版本策略**：
- MAJOR：原则被删除、重新定义或产生向后不兼容的变更。
- MINOR：新增原则或章节，或进行实质性内容扩展。
- PATCH：澄清说明、措辞修正、非语义性精化。

所有 Skill 和 PR 在合并前必须通过章程合规检查。偏离本章程原则的复杂度设计必须在 plan.md 的复杂度跟踪表中进行说明。
